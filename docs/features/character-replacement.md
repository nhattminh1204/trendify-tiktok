# Feature: Character Replacement Engine

## Purpose

Replaces the human subject in a viral template video with a selected AI Model.
The engine preserves the original video's motion patterns, camera angles, scene structure, and rhythm — only the visual identity (face and body appearance) of the human subject is replaced.

---

## Concept

```
Input:
  Viral template source video  +  AI Model (reference images)
      ↓
Process:
  1. Extract per-frame pose/motion data from source video
  2. Generate new video frames following extracted motion, using AI Model identity
  3. Composite new frames against original background
Output:
  New video — same movements, same scene structure — AI Model is the subject
```

---

## When This Engine Runs

This engine only runs when both conditions are true:
1. `VideoRenderJob.render_type = 'viral_template'`
2. `ViralTemplate.has_human_subject = true`

If `has_human_subject = false`, this job is skipped entirely — the source video passes directly to the Product Replacement Engine.

## Processing Pipeline

```
1. Load source video from MinIO
   (via viral_template → source_video_id → viral_video_sources.minio_url)

2. Pose Extraction
   — Run MediaPipe Pose on frames sampled at 8 fps
   — Extract 33 body keypoints per frame (x, y, visibility score)
   — Filter out frames where pose confidence < 0.7 (low confidence frames)
   — Store as pose_data JSON in MinIO

3. Identity Reference Preparation
   — Load AIModel.reference_image_urls (select up to 5 highest-quality images)
   — Pass reference images and pose_data to generation provider

4. Video Generation (via generation provider)
   — Provider generates new video following pose sequence
   — AI Model's face and body appear in each generated frame
   — Background is preserved from source video (or regenerated by provider)

5. Quality Validation
   — Duration check: output duration within ±2 seconds of source video
   — Face visibility: at least 60% of frames containing a subject must have a detectable face
   — If validation fails → increment retry_count, re-queue (max 2 retries)

6. Store output_video_url in MinIO
7. Publish CharacterReplacementCompleteEvent
```

## Pose Data Format (stored in MinIO)

```json
{
  "source_fps": 30,
  "sampled_fps": 8,
  "frame_count": 240,
  "keypoints_per_frame": [
    {
      "frame_index": 0,
      "timestamp_ms": 0,
      "confidence": 0.92,
      "keypoints": {
        "nose":           {"x": 0.52, "y": 0.18, "score": 0.99},
        "left_shoulder":  {"x": 0.42, "y": 0.35, "score": 0.98},
        "right_shoulder": {"x": 0.62, "y": 0.34, "score": 0.97},
        "left_hip":       {"x": 0.43, "y": 0.62, "score": 0.96},
        "right_hip":      {"x": 0.61, "y": 0.61, "score": 0.95}
      }
    }
  ]
}
```

Coordinates are normalized (0.0–1.0) relative to frame dimensions.

## Generation Providers

All providers are wrapped behind `ICharacterReplacementProvider` interface in `src/shared/`.

| Provider | Phase | Type | Notes |
|---|---|---|---|
| Kling AI API | Phase 1 primary | Cloud | Pose-conditioned video generation |
| Runway ML API | Phase 1 fallback | Cloud | Image-to-video with motion reference |
| ComfyUI + Wan2.1 | Phase 2 primary | Self-hosted, GPU | Open-source, full pose control |
| ComfyUI + AnimateDiff | Phase 2 alternative | Self-hosted, GPU | Lower VRAM than Wan2.1 |

Provider selection follows the same routing pattern as `AIProviderRouter` — primary → fallback on failure.

## Business Rules

1. Character replacement only runs when `ViralTemplate.has_human_subject = true` and `ai_model_id` is provided
2. If `ai_model_id` is null and `has_human_subject = true`, the render job is rejected at creation with `AI_MODEL_REQUIRED`
3. Hard timeout: generation provider call is killed after 10 minutes via `CancellationTokenSource`
4. Maximum 2 retries per job — after 2 failures: `status = 'failed'`, user notified
5. Output video must be within ±2 seconds of source video duration — otherwise treated as generation failure
6. AI Model must have `status = 'active'` at job creation time — rejected with `MODEL_NOT_AVAILABLE` otherwise
7. Pose data files (MinIO) are retained for 7 days then deleted by cleanup job
8. Output video is retained until the parent `VideoRenderJob` is archived by the user
9. Every generation provider call is logged to `AIUsageLog` with `feature_context = 'character-replacement'`
10. Concurrency limits per workspace: Free = 1, Pro = 2, Agency = 5 simultaneous jobs

## Background Jobs

### CharacterReplacementWorker
- Trigger: Job in `queued` status (Hangfire dedicated queue `character-replacement`)
- Concurrency: per-workspace slot enforcement (see Business Rule 10)
- Process: execute the 7-step Processing Pipeline above

### PoseDataCleanupJob
- Schedule: Daily at 02:00
- Process: Delete `pose_data_url` files from MinIO where `character_replacement_jobs.completed_at < NOW() - 7 days`

## Acceptance Criteria

### AC-1: Auto-Triggered
- [ ] When a viral_template render job is created with an AI Model, a CharacterReplacementJob is created within 5 seconds
- [ ] When template `has_human_subject = false`, no CharacterReplacementJob is created

### AC-2: Output Quality
- [ ] AI Model's face appears in at least 60% of frames containing a subject
- [ ] Generated movement visually matches the pose sequence of the source video
- [ ] Output video duration is within ±2 seconds of source video

### AC-3: Failure Handling
- [ ] Job auto-retries up to 2 times on provider error
- [ ] After 2 failures, render job detail page shows the error reason
- [ ] 10-minute timeout is enforced — a stalled provider call does not block the job indefinitely

### AC-4: Cost Tracking
- [ ] Every generation provider call logged to AIUsageLog
- [ ] Character replacement cost visible on render job detail page

## Implementation Notes

- Phase 1: Use Kling AI API. API endpoint: `POST https://api.klingai.com/v1/videos/image2video`
  - Pass: reference images array (from `ai_model.reference_image_urls`), motion reference video (source template), duration
  - Returns: async task ID → poll for completion
- Phase 1 fallback: Runway ML API. Endpoint: `POST https://api.dev.runwayml.com/v1/image_to_video`
- MediaPipe Pose runs in Python sidecar (same container as `ViralVideoAnalysisJob` — shared `ai-worker` Docker service)
- Output video must be re-encoded with FFmpeg to ensure consistent codec (H.264 + AAC, 1080×1920) regardless of provider output format
- `ICharacterReplacementProvider` in `src/shared/`:
  ```csharp
  public interface ICharacterReplacementProvider
  {
      Task<CharacterReplacementResult> GenerateAsync(
          string sourceVideoUrl,
          IEnumerable<string> referenceImageUrls,
          string poseDataUrl,
          int targetDurationSeconds,
          CancellationToken ct);
  }
  ```
- Pose data JSON is uploaded to MinIO path: `character-replacement/{job_id}/pose_data.json`
